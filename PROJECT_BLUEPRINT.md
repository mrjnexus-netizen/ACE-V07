--- سه اصل کلیدی استراتژی ---

اصل اول: تزریق اتمیک (Atomic Injection)
پروژه را در فازهای اتمیک به سیستم تزریق می‌کنیم تا ترمینال دچار خطا نشود و مسیر فایل‌ها

اصل دوم: تأیید مداوم (Continuous Verification)
بعد از هر prompt، عامل باید:
- typecheck اجرا کند (pnpm run typecheck)
- lint اجرا کند (pnpm run lint)
- تست‌های موجود اجرا کند
- صفر خطای runtime را تأیید کند
- integrity تمام imports را بررسی کند
اگر هر تأیید شکست خورد: STOP. همه تولیدات بیشتر متوقف می‌شوند تا مشکل حل شود.

اصل سوم: وفاداری معماری (Architecture Fidelity)
کد تولیدشده باید با معماری تعریف‌شده در این سند مطابقت کامل داشته باشد. هیچ معماری جایگزینی
اختراع نمی‌شود. هیچ الگوی رقیبی معرفی نمی‌شود. پایداری repository اولویت بالاتری از هر
تصمیم خلاقانه دارد.

--- مدیریت هوشمند توکن ---

مدیریت توکن برای عامل هوش مصنوعی بسیار مهم است. با این حال، مدیریت توکن نباید کیفیت کار

۱. اولویت‌بندی محتوا:
   - تولید فایل‌های کد کامل بر توضیحات اضافی مقدم است
   - گزارش وضعیت مختصر و دقیق باشد
   - از تکرار محتوای قبلاً تولیدشده پرهیز شود

۲. استراتژی تقسیم prompt:
   - هر prompt یک وظیفه اتمیک دارد
   - اگر توکن در میان اجرای یک فایل تمام شد، prompt بعدی باید از نقطه دقیق ادامه دهد
   - در ابتدای هر session جدید، وضعیت repository اسکن شود

۳. فشرده‌سازی زمینه:
   - اگر انقباض زمینه یا انحراف حافظه رخ داد: فایل‌های قبلی تولیدشده را مجدداً بخوانید
   - معماری را مجدداً تأیید کنید
   - exports و imports را مجدداً تأیید کنید
   - از وضعیت واقعی repository ادامه دهید
   - هرگز ماژول‌های پایدار موجود را متفاوت بازتولید نکنید

۴. پروتکل بازیابی:
   اگر کلید API تغییر کرد یا session جدیدی شروع شد:
   - این GOD file را مجدداً بخوانید
   - پوشه‌های موجود را اسکن کنید
   - از دقیقاً همانجایی که متوقف شدید ادامه دهید
   - زمینه را بازسازی نکنید — ادامه دهید

const applyTheme = (theme: ThemeConfig, locale: Locale) => {
  const root = document.documentElement;
  Object.entries(theme.variables).forEach(([key, value]) => {
    root.style.setProperty(key, value);
  });
  root.setAttribute('data-theme', theme.id);
  root.setAttribute('data-locale', locale);
  root.setAttribute('lang', locale);
  root.setAttribute('dir', 'ltr');
  applyLocaleTypography(locale, theme);
};

--- GLSL VERTEX SHADER ---

uniform float uTime;
uniform float uBassLevel;
uniform float uMidLevel;
uniform float uHighLevel;
uniform float uAudioData[128];
attribute float aScale;
void main() {
  vec3 pos = position;
  float noise = sin(pos.x * 8.0 + uTime * 0.5) *
                cos(pos.y * 8.0 + uTime * 0.3) *
                sin(pos.z * 8.0 + uTime * 0.4);
  float displacement = uBassLevel * 0.4 * noise;
  float spread = 1.0 + uMidLevel * 0.3;
  pos = pos * spread + normalize(pos) * displacement;
  vec4 mvPosition = modelViewMatrix * vec4(pos, 1.0);
  gl_Position = projectionMatrix * mvPosition;
  gl_PointSize = aScale * (uHighLevel * 3.0 + 1.5) *
                 (1.0 / -mvPosition.z);
}

--- GLSL FRAGMENT SHADER ---

uniform vec3 uColor;
uniform float uOpacity;
void main() {
  float dist = distance(gl_PointCoord, vec2(0.5));
  if (dist > 0.5) discard;
  float alpha = 1.0 - (dist * 2.0);
  gl_FragColor = vec4(uColor, alpha * uOpacity);
}

--- قرارداد انعطاف AI ---

همه call‌های AI باید exponential backoff retry پیاده‌سازی کنند (حداکثر ۳ تلاش، تأخیرها: 1s، 2s، 4s).
ceiling timeout باید روی هر call هوش مصنوعی خارجی اعمال شود.
حالت retry صف باید در Redis از طریق BullMQ persist شود.
job‌های شکست‌خورده باید بعد از restart سرور به صورت ایمن ادامه یابند.
شکست سرویس AI هرگز نباید thread اصلی اپلیکیشن را crash کند.

--- قوانین مدیریت توکن ---

قانون ۱: تولید کامل یا هیچ‌چیز
هر فایل کد باید به صورت کامل تولید شود. هیچ placeholder، هیچ TODO، هیچ
"implement similarly". اگر توکن برای تکمیل یک فایل کافی نیست، فایل را در prompt بعدی کامل کنید.
شروع کردن یک فایل و ناقص رها کردن آن مجاز نیست.

قانون ۲: اجتناب از تکرار
از تکرار محتوای قبلاً تولیدشده پرهیز کنید. در ابتدای هر session، فقط باید وضعیت repository
را بررسی کنید و از جایی که متوقف شدید ادامه دهید. بازتولید فایل‌های پایدار موجود ممنوع است.

قانون ۳: اولویت‌بندی هوشمند
در یک prompt با محدودیت توکن:
  - کد را بر توضیحات اولویت بدهید
  - پیاده‌سازی‌های بحرانی‌تر را اول بسازید
  - گزارش وضعیت را مختصر اما دقیق نگه دارید

قانون ۴: پروتکل از سر گرفتن
اگر session قطع شد:
  a. این GOD file را مجدداً بارگذاری کنید
  b. ساختار پوشه موجود را اسکن کنید
  c. آخرین فایل کامل‌شده را شناسایی کنید
  d. از prompt/مرحله بعدی ادامه دهید
  e. هرگز مراحل قبلی را تکرار نکنید مگر اینکه صریحاً خواسته شود

قانون ۵: مدیریت هوشمند context
- وقتی context پر می‌شود، ساختار معماری را بازخوانی کنید
- از تایپ‌های مشترک که قبلاً تعریف شده‌اند مجدداً استفاده کنید
- فقط تغییراتی را که واقعاً لازم هستند پیاده‌سازی کنید

--- وقتی خطا رخ می‌دهد ---

مرحله ۱: متوقف شوید
  - تولید کد جدید را متوقف کنید
  - خطای دریافت‌شده را کاملاً بخوانید
  - منبع خطا را شناسایی کنید

مرحله ۲: تشخیص دهید
  - خطا در کجای پروژه است؟ (فایل، خط)
  - آیا type error است؟ import error؟ runtime error؟
  - چه وابستگی‌هایی تحت تأثیر هستند؟

مرحله ۳: اصلاح کنید (surgical fix)
  - فقط جایی که خطا است را اصلاح کنید
  - هیچ refactoring اضافی انجام ندهید مگر لازم باشد
  - از pattern‌های معماری موجود پیروی کنید
  - بعد از اصلاح، کل pipeline تأیید را دوباره اجرا کنید

مرحله ۴: ادامه دهید
  - بعد از صفر خطا، به prompt بعدی بروید
  - خطای اصلاح‌شده را در گزارش وضعیت ذکر کنید

--- خطاهای رایج و راه‌حل ---

TypeScript Error - Type Mismatch:
  راه‌حل: فایل types/index.ts را بررسی کنید. interface را اصلاح کنید، نه کدی که از آن استفاده می‌کند.

Import Not Found:
  راه‌حل: ترتیب dependency generation را بررسی کنید. آیا فایل import‌شده قبلاً تولید شده؟

ESLint Warning:
  راه‌حل: هشدار را با اصلاح کد برطرف کنید. eslint-disable comment ممنوع است.

Hydration Mismatch:
  راه‌حل: بررسی کنید که آیا کد server-side و client-side متفاوت render می‌کنند. معمولاً
  به دلیل Math.random() یا Date.now() در render است.

Memory Leak در WebGL:
  راه‌حل: بررسی کنید که dispose() در useEffect cleanup صدا زده می‌شود.
